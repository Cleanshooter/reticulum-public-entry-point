# Reticulum GHCR image (reticulum-pep)

This repository builds and publishes a Docker image containing the Reticulum `rns` package and utilities. The image is intended to be consumed from GHCR (GitHub Container Registry) so users can pull and run the daemon without building or running python locally.

## Published image

- Image name: `ghcr.io/cleanshooter/reticulum-pep:<tag>` (also pushed as `:latest`)

## Examples

- Run the daemon with a named Docker volume (cross-platform, recommended):

```bash
docker volume create reticulum-data
docker run --rm -it \
	-v reticulum-data:/home/rnsuser/.reticulum \
	ghcr.io/cleanshooter/reticulum-pep:latest
```

- Run with a host bind mount (Linux / WSL / macOS):

```bash
mkdir -p ./reticulum-data
# ensure permissions for the container user if needed
docker run --rm -it \
	-v $(pwd)/reticulum-data:/home/rnsuser/.reticulum \
	ghcr.io/cleanshooter/reticulum-pep:latest
```

- Run detached and view logs:

```bash
docker run -d --name reticulum -v reticulum-data:/home/rnsuser/.reticulum ghcr.io/cleanshooter/reticulum-pep:latest
docker logs -f reticulum
```

- Pass arguments to `rnsd` (ENTRYPOINT is `rnsd`):

```bash
docker run --rm -it -v reticulum-data:/home/rnsuser/.reticulum ghcr.io/cleanshooter/reticulum-pep:latest --service
```

Docker Compose example

```yaml
version: '3.8'
services:
	reticulum:
		image: ghcr.io/cleanshooter/reticulum-pep:latest
		volumes:
			- reticulum-data:/home/rnsuser/.reticulum
		restart: unless-stopped

volumes:
	reticulum-data:
```

## Contributing

- If you want to improve the image (interfaces, additional utilities, config defaults), open a PR.

## Support & Notes

- The container runs `rnsd` as a non-root `rnsuser`. The entrypoint ensures the config directory is writable when using named volumes and common bind-mount workflows.
- If you need local development with a source checkout, I can add a `Dockerfile.dev` that copies local source into the image for iterative testing.

Questions or want me to add a `docker-compose.override.yml` example? Open a PR or tell me what example you prefer and I’ll add it.

Considerations for automation...

Reticulum's design philosophy inherently rejects centralization, even for security coordination. To collaboratively manage blackhole lists after an incident like yesterday's outage, volunteers can leverage Distributed Blackhole Management, a planned feature built on Network Identities. 

Trusted Peers, Not Central Authorities: The system will allow trusted network operators (identified by their Network Identities) to sign and publish their own blackhole lists.  Your node can then be configured to automatically subscribe to and enforce lists from these trusted peers. <- If this projects bootstrap config includes a trusted peers list for the config then it's no longer manual BUT then it become centralized... Well need an override mechanism... so people can manually udpate or opt-out of the autogenerated list. (maybe)
Reputation-Based Consensus: By subscribing to lists from multiple, reputable transport node operators, the network achieves a form of decentralized consensus on malicious actors, filtering spam or attackers without a single point of control. <- This is why I want an automated reputation assessment methodology. It's not feasible to scale the public network if everyone needs to know everyone else and manually trust them. This concept inherantly conflicts with the trustless network concept though. Plus manual trusts ultimately lead to stale interfaces... several transport node configurations I've reviewed already have lists of Nodes that have not been online for months. 
Immediate Action: For now, use the rnpath tool to manually add compromised identities to your local blackhole list (-B flag) and share these lists with other volunteers through out-of-band, trusted channels to coordinate a response. <- this is where we need to start with automation. Instead of manually adding bad actors we need rules like the ones here: https://pastebin.com/HXh2jSBc

Currently, Reticulum does not have an automated, built-in reputation system to solve the trust bootstrap problem for new transport node operators. 

Manual Trust is Required: Since trust is established manually new operators must identify and manually configure their nodes to trust existing, reputable backbone nodes by adding them to their configuration. There is no automatic metric or score (like uptime, bandwidth, or path reliability) that a new node can use to objectively assess whom to trust.
The Gap is Real: This creates a significant barrier to entry. A new node operator entering the public backbone space must rely on out-of-band methods—such as community forums, documentation, or personal introductions—to discover which nodes are trustworthy before they can effectively participate in distributed blackhole management or routing. 
Future Potential: While concepts like a reputation system are mentioned as future enhancements for projects built on Reticulum (e.g., Akita DDNS), and the "Zen of Reticulum" envisions "Distributed Blackhole Management" as a future feature, these systems are not yet implemented in the core Reticulum stack to automatically develop and distribute trust scores.

It's going ot be a tight road to strike a balance here... but this problem needs a solution from my perspective, so that's what I'm going to try and build.
